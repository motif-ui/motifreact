### Validation

Form provides a flexible validation system using the `validations` prop available in `<Form.Field>` and
`<Form.FieldGroup>` that actually manage the validations.

Please refer to **[Validation ↗](?path=/docs/form-validation--docs)** section for detailed information.

### Submitted Value Types

`onSubmit` callback returns the values of the form items in a single object. The keys of the object are the names of the
inputs. For those who uses typescript, it is essential to match the personal custom type with the one returned by the
onSubmit callback.

The type of the returned data object is `FormSubmitData` which is a generic type (that should extend
**`Record<string, any>`**). Using a generic type here is optional. When necessary, it can be used as follows:

```tsx
type CustomType = {
  name: string;
  age: number;
  city?: string;
};

const onSubmit = ({ values }: FormSubmitData<CustomType>) => {
  console.log(values.name);
  console.log(values.age);
  console.log(values.city);
};

<Form onSubmit={onSubmit}>
  <Form.Field name="name" label="Name"><InputText /></Form.Field>
  <Form.Field name="age" label="Age"><InputText /></Form.Field>
  <Form.Field name="city" label="City"><InputText /></Form.Field>
</Form>
```

It is also possible to use the generic type directly in the `<Form>` itself. Here is an example:

```tsx
type CustomType = {
  name: string;
  age?: number;
};

<Form<CustomType>
  onSubmit={data => console.log(data.values.name)}>

  <Form.Field name="name" label="Name"><InputText /></Form.Field>
  <Form.Field name="age" label="Age"><InputText /></Form.Field>
</Form>
```

### User Defined Errors

Besides the built-in `validations` system, `<Form>` accepts a declarative `externalErrors` prop to display error
messages that come from outside the form - for example after an async server response returns field-level errors, or to
show errors that are already known when the form is first displayed.

`externalErrors` is a plain object keyed by the `name` of a `<Form.Field>` or `<Form.FieldGroup>`, mapping to the error
message to display:

```tsx
type FormFields = { email: string; };
const [serverErrors, setServerErrors] = useState<Record<string, string>>({});

const onSubmit = async ({ values, isValid }: FormSubmitData<FormFields>) => {
  if (!isValid) return;

  const response = await submitToServer(values);
  if (!response.ok) {
    setServerErrors(response.fieldErrors); // e.g. { email: "This email is already taken" }
  }
};

<Form onSubmit={onSubmit} externalErrors={serverErrors}>
  <Form.Field name="email" label="Email"><InputText /></Form.Field>
</Form>
```

A few things to keep in mind:

- `externalErrors` is an external error layer. It renders exactly like a validation error but does **not** affect
  `validate()`'s `isValid` result on the next submit. It's up to the consumer to decide what a resubmission should do.
- An error set through `externalErrors` is cleared automatically as soon as the user edits that field, the same way a
  validation error would be.
- Errors are keyed at the same granularity as `validations`/`groupValidations` - i.e. by the name of the
  `<Form.Field>`/`<Form.FieldGroup>`, not by individual inputs nested inside a group.
- By default the form does **not** clear its values after a submit and values only get cleared when you set
  `resetIfValidatedOnSubmit`. If you use `resetIfValidatedOnSubmit` together with `externalErrors`, be aware that
  `resetIfValidatedOnSubmit` fires synchronously right after a client-side-valid submit, before an `await`ed server call
  can resolve. So, avoid combining `resetIfValidatedOnSubmit` with an async `onSubmit` that sets `externalErrors` from
  the response, since the values will already be cleared by the time the error arrives.
- If a field currently has its own internal error (e.g. Upload's own file validation), an `externalErrors` update for
  that same field is held back until the internal error clears - the internal one stays on screen since it's the more
  immediately-actionable problem, and it's what's actually keeping the field invalid.

### Use with `<Grid>`

Using **Form.Field** directly inside **Form** is the easiest way to create a form with the default layout order. There is
also a layout customization option using the `<Grid>` component inside **Form**.

```tsx
<Form>

  <Grid fluid>
    <Row>
      <Col size={3}>
        <Form.Field name="name" label="Name">
          <InputText />
        </Form.Field>
      </Col>
      <Col size={3}>
        <Form.Field name="surname" label="Surname">
          <InputText />
        </Form.Field>
      </Col>
      <Col size={3}>
        <Form.Field name="age" label="Age">
          <InputText />
        </Form.Field>
      </Col>
    </Row>
    <Row>
      <Col size={6}>
        <Form.Field name="city" label="City">
          <InputText />
        </Form.Field>
      </Col>
      <Col size={6}>
        <Form.Field name="country" label="Coutry">
          <InputText />
        </Form.Field>
      </Col>
    </Row>
  </Grid>

</Form>
```

> **Note:** For more please refer to [Form](?path=/docs/form-form--docs)
